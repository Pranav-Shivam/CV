Path Search vs Local Search
The fundamental shift in how we think about solutions

Sessions 2–4 cared about the journey — the path from initial state to goal, its cost, and the sequence of steps. Session 5 introduces a completely different philosophy: only the destination matters.

Fig 1 — Path Search tracks every step; Local Search only keeps the current configuration.

TermMeaningExam one-liner
Feasible SolutionAny valid state — queens placed, constraints obeyed, may still conflict“An answer, not necessarily optimal”
Global OptimumThe single best feasible solution — fitness = maximum (f=6 for 4-Queens)“The best possible answer”
Local OptimumState where all neighbours are worse — not necessarily globally best“Stuck, no immediate improvement”
Fitness ValueNumerical score measuring how good a state is — computed from the state“Computed from the state itself”
♝
N-Queens analogy: Path search starts with an empty board and places queens one by one, tracking each choice. Local search starts with all queens already placed randomly and asks: “can I improve this board by moving exactly one queen?”

1.2 — When to Use Local Search

ConditionExplanationReal-world example
Path doesn’t matterOnly the final configuration is needed, not how you got thereN-Queens, job scheduling, timetabling
State space is hugePath search (BFS/A*) runs out of memory — O(bᵈ) is infeasibleHyperparameter tuning: millions of combinations
Any valid state is a startGenerate a random feasible solution instantly and improve itRandomly placed queens → move to reduce conflicts
Optimisation over satisfactionGoal is to maximise/minimise a score, not reach a fixed goal nodeMinimise energy, maximise non-conflicting pairs

1.3 — The State IS the Solution

In local search, any valid state is a “feasible solution.” This does not mean it is optimal — it means you have something to evaluate and improve. The fitness value tells you how good the current state is. The algorithm climbs toward higher fitness until no improvement is possible — at which point you have found a local optimum, which may or may not coincide with the global one.

02
State Representation & Neighbours
Vector notation and the 1-NN rule for 4-Queens

A 4-Queens board state is encoded as a vector [V1, V2, V3, V4] where each value is the row position of the queen in that column (1 = top, 4 = bottom).

Fig 2 — State [2,4,2,2]: columns 3 & 4 share row 2 (conflict shown in red). 1-NN generates 12 neighbours.

⚠
Don’t confuse with ML’s K-Nearest Neighbours! In local search, 1-NN means “change 1 queen at a time.” In machine learning, KNN is a classification algorithm. Same abbreviation, completely different concept.
NEIGHBOURHOOD SIZES1-NN: change 1 queen → N×(N−1) neighbours e.g. 4×3 = 12 2-NN: change 2 queens → C(N,2)×(N−1)² e.g. 6×9 = 54 K-NN: change K queens → exponentially many combinations Exam rule: ALWAYS use 1-NN unless explicitly stated otherwise.
03
Fitness Function
Live calculator — set queen positions and watch the math

Fitness vs Heuristic — key difference: A heuristic h(n) estimates cost to goal from state n — given externally, used in path search (A*, Greedy). A fitness f(s) scores how good a state is — computed directly from the state itself, used in local search. There is no predefined goal node; fitness replaces both the goal test and the heuristic.

📐
Diagonal conflict rule: Queens at columns c1, c2 in rows r1, r2 conflict diagonally if |r1 − r2| = |c1 − c2|. Same row conflict if r1 = r2. Same column is impossible by design (one queen per column).

04
Hill Climbing — The Algorithm
Greedy local improvement until stuck at a peak

Fig 3 — Hill Climbing always reaches a nearby peak, but which peak depends entirely on the starting position.

ALGORITHMHILL CLIMBING Step 1. Pick a RANDOM valid state s ← feasible solution Step 2. Evaluate fitness f(s) ← count non-conflicting pairs Step 3. Generate all 1-NN neighbours of s ← 12 for 4-Queens Step 4. Evaluate f(n) for all neighbours n Step 5. IF any neighbour n has f(n) > f(s): → s = best_neighbour — GO TO Step 3 Step 6. IF no neighbour improves on f(s): → STOP. Return s ← stuck at local maximum

Worked Example — 4-Queens

1Pick random state [1,4,2,2]. Compute fitness = 4 non-conflicting pairs. Goal is f = 6.
2Generate 12 neighbours. Evaluate each. Best neighbour: [1,4,3,2] with f = 5. Since 5 > 4, move there.
3Now at [1,4,3,2] with f = 5. Generate its 12 neighbours. All have f ≤ 5 — no improvement possible.
4STOP. Return [1,4,3,2] with f = 5. Goal was f = 6 — stuck at local optimum. Not solved.
⚡
The core limitation: Hill Climbing guarantees only a local optimum. Whether that equals the global optimum depends entirely on the starting state. This is what motivates Random Restart and Beam Search.
05
Three Variants of Hill Climbing
Strategies to escape local optima and explore more of the landscape

Always picks the best neighbour. Stops the moment no improvement exists. Simple and fast, but very prone to getting stuck at local optima.

Cannot escape local optimaMemory: O(1)Single run

When stuck at a local maximum, pick a completely new random starting state and run Hill Climbing again. Repeat K times, keep the best result found.

K = 1: Same as basic HC K = 5: Run HC 5 times, return best result K = ∞: Guaranteed to find global optimum (no time bound)
Explores multiple landscape areasSequential — not parallelK = number of restarts

Instead of always picking the best neighbour, compute a probability for each fitness value proportional to how often it appears, then select probabilistically. Helps escape near-plateau regions.

Example with 12 neighbours: f=1: appears 2 times → P = 2/12 = 0.17 f=2: appears 5 times → P = 5/12 = 0.42 ← highest — SELECT this group f=3: appears 4 times → P = 4/12 = 0.33 f=4: appears 1 time → P = 1/12 = 0.08 → Pick the group with highest probability; random among ties.

Beam Search — Parallel Hill Climbing
K independent threads; first to find global optimum broadcasts STOP

Beam Search = K threads of Hill Climbing running simultaneously. If any thread finds the global optimum, it broadcasts STOP to all others. Unlike Random Restart, the threads run in parallel — not one after another.

Fig 4 — K=3 parallel threads. Thread 1 finds f=6 and broadcasts STOP; threads 2 & 3 halt immediately.

ALGORITHMBEAM SEARCH (K parallel threads) Step 1. Start K random states simultaneously: s₁, s₂, ..., sK Step 2. Each thread independently applies Hill Climbing: Thread 1: s₁ → generate neighbours → evaluate → move to best Thread 2: s₂ → generate neighbours → evaluate → move to best ... Step 3. IF any thread finds global optimum: → Broadcast STOP to all other threads → Return the winning state Step 4. IF no thread finds global optimum (all stuck): → Stochastic variant: each picks next state by probability → Repeat from Step 2 KEY: Threads do NOT share information mid-search. They only communicate when ONE finds the global answer. K = 1 → Beam Search = Stochastic Hill Climbing

Interactive Simulation

Random Restart vs Beam Search:
Random Restart runs K hill climbs one after another — total time = K × one HC run.
Beam Search runs K hill climbs simultaneously — total time ≈ 1 HC run.
With K processors available, Beam Search is approximately K times faster.

Stochastic Beam Search

Standard Beam Search: each thread picks the best neighbour (greedy, like Basic HC).

Stochastic Beam Search: each thread picks a neighbour probabilistically (like Stochastic HC). This adds exploration and helps threads avoid converging to the same local optimum.

VariantSelectionEscape local optima
Beam SearchGreedy — always bestSomewhat (K threads)
Stochastic Beam SearchProbabilistic by frequencyBetter — probabilistic + parallel

The Optimization Landscape
Visualising why local search behaves the way it does

Think of fitness as a hilly terrain. Each state is a point on the surface; its height is its fitness. Hill Climbing is like climbing this terrain blindfolded — it can only feel the slope immediately around the current position.

Fig 5 — The optimization landscape. Local max at f=4, global max at f=6. Start A gets stuck; Start B reaches the global.

FeatureDescriptionAlgorithm implication
Local MaximumAll neighbours are worse — algorithm stops hereBasic HC always stops here; may not be global
Global MaximumThe single highest point across all statesThe target; f=6 for 4-Queens
PlateauFlat region — all neighbours have equal fitnessHC gets confused; stochastic variants handle better
RidgeNarrow elevated strip — movement along it is trickyHard for HC; needs diagonal moves it cannot make
ValleyLocal minimum (when minimising fitness)Same as local max when minimising

09
Preview — Genetic Algorithms
What changes when threads start cooperating instead of working alone

The key distinction between Beam Search and Genetic Algorithms is whether the threads share information mid-search.

AspectBeam SearchGenetic Algorithms
Starting pointK random independent statesInitial population of states
Thread collaborationNone — work independentlyYes — states share information (crossover)
Information sharingOnly final answer sharedIntermediate states combine to produce children
Variation mechanismMove one queen (1-NN)Crossover (combine parents) + Mutation
InspirationParallel computingBiological evolution and natural selection
🧬
The key insight: In Beam Search, K threads are like K people solving the same puzzle separately in different rooms. In Genetic Algorithms, they cooperate — like parents having children who inherit features from both. The “fittest” states reproduce; weaker ones are discarded. This biological metaphor makes GAs significantly more powerful for certain problem types.

Exercise 1Hill Climbing Trace on 4-Queens
▼
Starting state: [3, 1, 4, 2] — queen in col1=row3, col2=row1, col3=row4, col4=row2
aCalculate fitness using non-conflicting pairs. Show ALL 6 pair checks.
bGenerate all 12 1-NN neighbours.
cCalculate fitness for each neighbour.
dWhich neighbour does Hill Climbing move to?
eFrom that neighbour, are you closer to the global optimum f=6?
Solution
State [3,1,4,2]: Q1=row3, Q2=row1, Q3=row4, Q4=row2 All 6 pair checks (diagonal rule: |r1−r2| = |c1−c2|) Q1,Q2: rows 3 vs 1 → |3−1|=2, |col1−col2|=1 → 2≠1 → NOT conflicting ✓ Q1,Q3: rows 3 vs 4 → |3−4|=1, |1−3|=2 → 1≠2 → NOT conflicting ✓ Q1,Q4: rows 3 vs 2 → |3−2|=1, |1−4|=3 → 1≠3 → NOT conflicting ✓ Q2,Q3: rows 1 vs 4 → |1−4|=3, |2−3|=1 → 3≠1 → NOT conflicting ✓ Q2,Q4: rows 1 vs 2 → |1−2|=1, |2−4|=2 → 1≠2 → NOT conflicting ✓ Q3,Q4: rows 4 vs 2 → |4−2|=2, |3−4|=1 → 2≠1 → NOT conflicting ✓ Fitness = 6 / 6 ← GLOBAL OPTIMUM! [3,1,4,2] is already a solution. Hill Climbing stops immediately at Step 2. Parts (b)–(e) are not needed. The random starting state happened to be the global optimum directly.
💡
This illustrates why Random Restart can find the global optimum in very few restarts — sometimes the initial random state is already at or near the global optimum.
Exercise 2Local Beam Search with K=3
▼
Three starting states: [2,4,1,3] f=5   [1,3,1,4] f=3   [4,2,3,1] f=4
aWhich thread is most promising?
bThread 1 generates neighbours and best has f=6. What happens?
cIf no thread found f=6 in round 1, describe round 2.
dIf K=1, what algorithm does Beam Search become?
Solution
(a) Most promising thread: Thread 1 [2,4,1,3] with f=5 — highest fitness among the three. Threads 2 and 3 have f=3 and f=4 respectively. (b) Thread 1 finds f=6: → Thread 1 finds global optimum f=6 → Thread 1 broadcasts STOP to ALL other threads → Threads 2 and 3 halt immediately, regardless of their current state → Beam Search terminates; returns the winning state from Thread 1 (c) Round 2 if no winner in round 1: → Each thread independently generates 12 neighbours → Each thread moves to its best neighbour (or probabilistic if stochastic variant) → All 3 threads continue climbing in parallel → Repeat until a thread hits f=6 or iteration limit is reached (d) K=1 reduces to: K=1 → Stochastic Hill Climbing One thread, probabilistic selection = Stochastic HC by definition.
Exercise 3Stochastic HC — Probability Calculation
▼
Starting state: [2,2,4,3] f=3. After generating 12 neighbours:
f=1: ×1  |  f=2: ×2  |  f=3: ×5  |  f=4: ×3  |  f=5: ×1
aCalculate probability for each fitness value.
bWhich fitness group has the highest probability?
cIf two neighbours have f=4, which one do you pick?
dHow does this differ from Basic Hill Climbing?
Solution
(a) Probabilities (frequency ÷ 12): P(f=1) = 1/12 = 0.083 ( 8.3%) P(f=2) = 2/12 = 0.167 (16.7%) P(f=3) = 5/12 = 0.417 (41.7%) ← highest probability group P(f=4) = 3/12 = 0.250 (25.0%) P(f=5) = 1/12 = 0.083 ( 8.3%) Sum = 12/12 = 1.000 ✓ (b) Highest probability group: f=3 with P=0.417 — Stochastic HC selects from this group. This is NOT the highest fitness group! Basic HC would pick f=5. (c) Tie within the selected group: Pick uniformly at random among the neighbours in the winning group. (d) Difference from Basic HC: Basic HC: moves to neighbour with MAXIMUM fitness (f=5 here). Stochastic HC: moves to neighbour from MOST PROBABLE group (f=3 here). Stochastic HC may move to a LOWER fitness state — intentional exploration.
ChallengeExercise 4 — Hyperparameter Tuning as Local Search
▼
Tuning a neural network: lr ∈ {0.001, 0.01, 0.1}   batch ∈ {16, 32, 64}   epochs ∈ {10, 20, 30}
aWhat is the total state space size?
bDefine a fitness function for this problem.
cStarting from [lr=0.01, batch=32, epochs=20], how many 1-NN neighbours exist?
dWhich local search variant would you recommend, and why?
eWhat is the “global optimum” in this context?
Solution
(a) Total state space: 3 × 3 × 3 = 27 possible configurations (b) Fitness function: f(state) = validation accuracy of the trained model Higher accuracy = better state. Goal: maximise f. (c) 1-NN neighbours from [lr=0.01, batch=32, epochs=20]: lr: 0.01→0.001 | 0.01→0.1 = 2 neighbours batch: 32→16 | 32→64 = 2 neighbours epochs: 20→10 | 20→30 = 2 neighbours Total = 6 neighbours (d) Recommended algorithm: Beam Search (K=3 to 5) Each evaluation trains a full model — expensive. Beam Search gives K parallel explorations with far fewer total evaluations than grid search. (e) Global optimum: The hyperparameter combination with the highest validation accuracy. The true global is unknown; local search approximates it efficiently.
💡
This is how real ML practitioners tune large models — exhaustive grid search over all 27 combinations is wasteful when each evaluation is expensive. Local search finds a good solution with far fewer evaluations.
